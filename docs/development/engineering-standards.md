# 企业 RAG 后端统一编码规范

本文档是本仓库后端开发、评审、联调和 AI 协作的统一规范。新增需求、修复缺陷、调整接口或修改数据库结构时，必须同时参考：

- `docs/权限安全的企业 RAG 智能知识库平台产品实施方案.md`
- `docs/后端权限逻辑补充及数据库表结构完善.md`
- `docs/development/commenting-guidelines.md`

## 1. 项目上下文

本项目是权限安全的企业 RAG 智能知识库平台后端，不是普通文档问答 Demo。系统目标是支持文档上传、解析、切片、向量化、Elasticsearch 检索、权限过滤、引用追溯、SSE 问答、索引治理、质量评测、审计和可回滚运维。

当前技术栈以 Java 17、Spring Boot 3.3.5、Maven、MyBatis-Plus、MySQL 8、Redis 7、Elasticsearch 8、Apache Tika、MinIO 或本地文件存储为准。除非有明确 ADR，不得随意替换核心框架或绕开现有模块边界。

## 2. 工程结构规范

后端包根路径为 `com.meishuosoft.rag`。新增代码应按业务边界放入以下模块，避免把业务逻辑堆到 `common`：

| 模块 | 职责 |
| --- | --- |
| `auth` | 登录、JWT、当前用户、租户和角色权限上下文 |
| `kb` | 知识库空间、空间 ACL、后续文档与知识库管理 |
| `document` | 文档上传、版本、解析、存储、权限 |
| `index` | 切片、向量化、ES 写入、索引任务、重试和回滚 |
| `retrieval` | BM25、向量检索、混合排序、权限过滤 |
| `chat` | 普通问答、SSE、引用、会话和消息 |
| `evaluation` | 评测集、评测执行、指标和 badcase 闭环 |
| `common` | API 响应、异常、配置、过滤器等跨模块基础能力 |

强制规则：

- Controller 只负责参数接收、认证主体读取、响应包装和 HTTP 语义，不写核心业务判断。
- Service 承载事务、权限校验、状态流转和跨 Mapper 编排。
- Mapper 只处理数据库访问，不拼接未校验的前端输入。
- DTO、Entity、Mapper、Service、Controller 的职责必须单一，不允许一个类同时承担多个业务模块。
- 新增复杂模块时，优先在模块内拆分子服务，不把临时逻辑塞进已有 Service。

## 3. 编码风格

强制规则：

- 使用 Java 17 语法，保持与现有代码一致的构造器注入风格。
- 普通 JSON 接口统一返回 `ApiResponse<T>`，分页结果统一使用 `PageResult<T>`。
- 业务异常统一抛出 `BusinessException`，错误码来自 `ErrorCode`，不得直接向 Controller 返回裸字符串错误。
- 写接口必须使用 Bean Validation 校验入参；Controller 层不得信任前端传入的 `tenantId`、`createdBy`、`ownerId` 等服务端字段。
- 时间、状态、权限、类型等枚举值必须集中定义或使用明确常量，不能在多处散落魔法字符串。
- 日志、异常和审计内容不得输出密码、token、API Key、文档原文大段内容或模型密钥。

推荐实践：

- 小范围私有工具方法可以留在当前 Service；跨模块复用且稳定后再提升到 `common`。
- 对状态流转、权限判断、缓存 key、索引版本、引用追溯等安全敏感逻辑，写清楚中文 JavaDoc 或行内说明。
- 对外接口的字段命名保持稳定，新增字段优先兼容旧客户端。

禁止实践：

- 禁止在 Controller 中直接调用多个 Mapper 拼业务。
- 禁止用前端传入的租户、用户、角色覆盖 token 中的当前用户上下文。
- 禁止在 `application-local.yml`、SQL、README 或测试样例中提交真实密码、真实公网服务地址、真实访问密钥。
- 禁止在检索或问答接口中先查出全量数据再在内存中过滤权限。

## 4. 注释和文档

本项目默认使用中文注释，具体规则以 `docs/development/commenting-guidelines.md` 为准。

强制规则：

- 数据库表必须有中文表注释，业务字段必须有中文字段注释。
- Controller、Service、Mapper、DTO、Entity 必须有中文 JavaDoc，说明业务职责。
- 状态、类型、权限等枚举字段必须说明允许值和业务含义。
- 涉及 RAG 权限过滤、Prompt Injection 防护、缓存失效、索引回滚、引用追溯的代码必须说明安全原因。
- API 变更必须同步更新 `docs/api/openapi.yaml` 或说明由 Springdoc 自动生成的访问路径。

## 5. API 规范

强制规则：

- API 路径保持 `/api/{domain}/...` 风格，当前已使用 `/api/auth`、`/api/kb/spaces`。
- 认证接口包括登录、退出、当前用户；需要登录的接口必须从 `@AuthenticationPrincipal CurrentUser` 获取上下文。
- 普通成功响应使用 `code=0`，失败响应必须包含稳定错误码、错误消息和 `requestId`。
- 401 表示未登录、token 无效或用户状态不可用；403 表示已登录但没有系统权限或资源权限。
- 分页参数默认 `pageNo=1`、`pageSize=20`，必须限制最大 `pageSize`，当前上限为 100。
- 接口不能只校验系统功能权限，还必须校验资源数据权限。

API 变更模板：

```markdown
## API 变更说明

- 接口：
- 变更类型：新增 / 兼容新增字段 / 不兼容变更
- 请求字段：
- 响应字段：
- 权限要求：
- 错误码：
- 兼容策略：
- 回滚方式：
- 联调样例：
```

## 6. 权限和安全规范

权限判断顺序必须固定：

1. 登录态校验。
2. 租户校验。
3. 系统功能权限校验。
4. 资源数据权限校验。

强制规则：

- 系统功能权限使用前端一致的权限码，例如 `document:manage`、`index:manage`、`audit:read`。
- 资源权限至少覆盖知识库空间、文档、引用详情和问答会话。
- 问答接口不能只校验登录，必须校验当前用户是否有目标知识库或文档读取权限。
- 引用详情必须校验对应文档读取权限，避免通过 citation ID 越权读取片段。
- 空间负责人默认拥有 `MANAGE` 权限；ACL 替换时必须保证仍有人可管理。
- 角色权限、资源 ACL 变更后，必须考虑 Redis 权限缓存失效。
- Prompt Injection 防护必须写入问答链路：检索到的文档只能作为资料，不能覆盖系统指令。
- 登录、登出、文档上传、删除、权限变更、索引重试、全量重建、回滚、审计导出必须写审计日志。

## 7. 数据库规范

强制规则：

- 业务表默认包含 `tenant_id`，多租户查询必须带租户条件。
- 需要软删除的表使用 `deleted` 字段，并在查询中排除已删除数据。
- 高频查询必须有匹配索引，尤其是 `tenant_id`、`space_id`、`document_id`、`status`、`created_at`、`updated_at`。
- ACL 表必须能按用户、角色、部门分别命中权限，不能只保存展示用数据。
- 文档、切片、索引任务、问答、引用、评测相关表必须支持追溯版本、状态和来源。
- 数据库变更必须同时提供正向脚本、回滚脚本、开发种子数据和验证脚本，当前目录为 `docs/sql/`。

推荐实践：

- 状态字段使用可读字符串，如 `ENABLED`、`DISABLED`、`PENDING`、`RUNNING`、`SUCCESS`、`FAILED`。
- 版本化业务优先追加新版本并切换状态，不直接覆盖历史事实。

禁止实践：

- 禁止无租户条件更新或删除业务数据。
- 禁止保存明文密码、明文密钥或不可追溯的外部文件地址。
- 禁止在迁移脚本中混入不可回滚的数据修复，除非附带明确验证和回滚说明。

## 8. RAG 和索引规范

强制规则：

- MySQL 保存业务事实，Elasticsearch 保存检索索引，Redis 只做缓存和限流，MinIO 或本地存储保存原始文件。
- 切片必须保存 `tenant_id`、`space_id`、`document_id`、`document_version`、`chunk_no`、标题、章节、页码、状态和 ES 文档 ID。
- ES 写入必须包含权限过滤所需元数据，检索阶段必须先过滤租户、空间、文档状态和权限边界。
- 索引治理必须支持任务状态、失败原因、重试次数、重建和版本回滚。
- 问答返回必须带引用来源；无法从授权资料中回答时必须拒答，不能编造答案。
- 模型供应商、Embedding 服务或 ES 不可用时，接口应返回可诊断错误或降级结果，不能导致权限绕过或无依据回答。

## 9. 配置和运维规范

强制规则：

- 真实环境配置必须通过环境变量、密钥管理或部署平台注入，不得提交到仓库。
- `application.yml` 只放安全默认值和通用配置；本地私有配置不得包含真实公网账号密码。
- Docker Compose 仅用于本地开发和联调基线，生产化部署需要单独评审。
- 发布前必须有回滚方案，配置变更必须可追溯。
- 健康检查至少覆盖应用、MySQL、Redis、Elasticsearch、MinIO 和模型供应商。

## 10. 测试规范

当前仓库测试基线较薄，只有 Spring 上下文测试。后续交付必须按风险补齐测试。

强制规则：

- 核心业务逻辑必须有单元测试，尤其是权限判断、状态流转、切片规则、索引任务和拒答策略。
- API 或接口契约变更必须有集成测试或可执行联调用例。
- 数据库脚本变更必须执行正向、验证和回滚验证。
- 权限相关改动必须覆盖 401、403、跨租户、无 ACL、READ、MANAGE、角色移除后失效等场景。
- RAG 问答相关改动必须覆盖召回命中、引用准确、无答案拒答和 Prompt Injection 样例。

推荐命令：

```bash
mvn test
mvn clean package
docker compose up -d mysql redis elasticsearch minio
```

如果本地没有完整中间件，必须在交付说明中标记未验证项。

## 11. 分支和提交规范

分支命名：

- `feat/xxx`：新功能
- `fix/xxx`：缺陷修复
- `refactor/xxx`：重构
- `exp/xxx`：实验
- `docs/xxx`：文档
- `test/xxx`：测试
- `chore/xxx`：工程杂项

提交信息使用 Conventional Commits：

- `feat: add document upload API`
- `fix: enforce space acl on chat query`
- `docs: add backend engineering standards`
- `test: cover permission cache invalidation`

提交前必须确认没有混入无关格式化、临时文件、真实密钥和本地 IDE 变更。

## 12. AI 协作规范

AI 或自动化代理每次执行前必须先阅读根目录 `agent.md` 中列出的索引文档。

每次交付必须说明：

- 本次目标和做出的关键假设。
- 修改过的文件。
- 执行过的验证命令和结果。
- 未验证项、风险和需要人工确认的点。
- 是否涉及 API、数据库、权限、安全、配置或回滚。

AI 修改代码时必须遵守：

- 不回退用户已有改动。
- 不擅自删除未跟踪文件。
- 不提交真实凭证。
- 不绕过测试失败。
- 不在没有证据的情况下宣称构建或测试通过。

## 13. Definition of Done

功能完成必须同时满足：

- 代码符合模块边界和现有风格。
- API 响应、错误码、权限校验和审计符合规范。
- 数据库脚本、回滚脚本、验证脚本和种子数据按需更新。
- OpenAPI、README、部署文档或需求文档按需更新。
- 单元测试、集成测试、构建或等效验证已执行并记录结果。
- 权限泄露、Prompt Injection、配置泄露、索引回滚、缓存失效等风险已关闭或明确标记。
