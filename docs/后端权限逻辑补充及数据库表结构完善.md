# 后端权限逻辑补充及数据库表结构完善

版本：V1.0  
日期：2026-07-01  
适用项目：权限安全的企业 RAG 智能知识库平台  
适用后端：Java / Spring Boot / Spring Security / MySQL

---

## 1. 背景

当前前端已经接入基于 `permission` 字符串的权限控制：

1. 菜单根据当前用户 `permissions` 过滤。
2. 路由根据 `meta.permission` 判断是否允许访问。
3. 按钮根据 `hasPermission(permission)` 控制可操作性。
4. 接口返回 401 时跳转登录。
5. 接口返回 403 时展示无权限状态。

前端权限只负责用户体验。后端必须同步实现接口鉴权、资源权限校验和数据库权限模型，否则用户仍可绕过前端直接请求接口。

---

## 2. 权限模型

后端需要实现两层权限：

| 类型 | 说明 | 示例 |
|---|---|---|
| 系统功能权限 | 控制菜单、路由、按钮、接口操作能力 | `document:manage`、`index:manage` |
| 资源数据权限 | 控制用户是否能访问某个知识库、文档、引用内容 | 用户是否可读 `spaceId`、`documentId` |

权限判断顺序建议：

```text
登录态校验
  -> 租户校验
  -> 系统功能权限校验
  -> 资源数据权限校验
  -> 业务状态校验
  -> 执行业务逻辑
```

---

## 3. 前端当前依赖的权限点

| 权限码 | 用途 |
|---|---|
| `kb:manage` | 知识库管理菜单和页面 |
| `document:manage` | 文档管理、上传、删除、构建索引、权限配置 |
| `document:read` | 文档详情、引用详情、文档预览 |
| `index:manage` | 索引治理、失败重试、全量重建 |
| `eval:manage` | 评测集、评测执行、badcase 处理 |
| `eval:read` | 评测报告查看 |
| `audit:read` | 审计日志查看 |

后端返回给前端的 `permissions` 必须使用以上字符串，保持大小写和冒号格式一致。

## 4. 必须提供的认证接口

## 4.1 登录接口

```http
POST /api/auth/login
Content-Type: application/json
```

请求：

```json
{
  "tenantCode": "demo",
  "username": "admin",
  "password": "123456"
}
```

响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "token": "jwt-token",
    "userInfo": {
      "id": "1",
      "tenantId": "1",
      "username": "admin",
      "realName": "管理员",
      "departmentId": "1",
      "roles": ["ADMIN"],
      "permissions": [
        "kb:manage",
        "document:manage",
        "index:manage",
        "eval:manage",
        "audit:read"
      ]
    }
  }
}
```

后端要求：

1. 校验租户是否存在。
2. 校验用户状态是否启用。
3. 校验密码。
4. 生成 token。
5. 聚合用户角色和权限。
6. 写入登录审计日志。

## 4.2 当前用户接口

```http
GET /api/auth/me
Authorization: Bearer {token}
```

响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "id": "1",
    "tenantId": "1",
    "username": "admin",
    "realName": "管理员",
    "departmentId": "1",
    "roles": ["ADMIN"],
    "permissions": [
      "kb:manage",
      "document:manage",
      "index:manage"
    ]
  }
}
```

后端要求：

1. token 无效或过期返回 401。
2. 用户禁用返回 401 或业务错误码。
3. 必须返回 `permissions: string[]`。
4. 权限数据建议可缓存，但角色权限变更后需要失效。

## 4.3 退出登录接口

```http
POST /api/auth/logout
Authorization: Bearer {token}
```

说明：

1. JWT 无状态模式下可只记录审计日志。
2. 如果使用 Redis token，应删除当前 token。
3. 如果使用 refresh token，应同时失效 refresh token。

## 5. Java 鉴权逻辑建议

## 5.1 权限注解

建议新增方法级权限注解：

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RequirePermission {
    String value();
}
```

使用示例：

```java
@RequirePermission("document:manage")
@PostMapping("/api/kb/spaces/{spaceId}/documents")
public ApiResponse<Void> uploadDocument(...) {
    return ApiResponse.ok();
}
```

也可以使用 Spring Security：

```java
@PreAuthorize("@permissionService.hasPermission(authentication, 'document:manage')")
```

## 5.2 系统权限服务

```java
public interface PermissionService {

    boolean hasPermission(Long userId, String permissionCode);

    Set<String> listUserPermissions(Long userId);

    boolean hasAnyPermission(Long userId, Collection<String> permissionCodes);
}
```

实现要求：

1. 根据 `user_id -> role_id -> permission_id` 查询权限。
2. 必须带 `tenant_id` 过滤。
3. 忽略已删除角色、权限和用户。
4. 用户禁用时不返回权限。
5. 可将用户权限缓存到 Redis。

## 5.3 资源权限服务

```java
public interface ResourcePermissionService {

    boolean canAccessSpace(Long userId, Long spaceId);

    boolean canManageSpace(Long userId, Long spaceId);

    boolean canReadDocument(Long userId, Long documentId);

    boolean canManageDocument(Long userId, Long documentId);
}
```

资源权限判断来源：

1. 用户直接授权。
2. 用户所属角色授权。
3. 用户所属部门授权。
4. 知识库负责人默认有管理权限。
5. 系统管理员可拥有全局管理权限。

## 7. 错误码约定

| HTTP 状态 | 业务 code | 说明 |
|---|---|---|
| 401 | 401 | 未登录、token 无效、token 过期 |
| 403 | 403 | 已登录但没有权限 |
| 404 | 404 | 资源不存在 |
| 409 | 409 | 业务状态冲突，例如非失败任务不能重试 |
| 500 | 500 | 系统异常 |

响应示例：

```json
{
  "code": 403,
  "message": "没有权限访问该资源",
  "data": null,
  "requestId": "req_202607010001"
}
```

## 8. 数据库表结构

以下 DDL 以 MySQL 为例。主键可使用雪花 ID、数据库自增或团队统一 ID 方案。

## 8.1 用户表

```sql
CREATE TABLE sys_user (
  id BIGINT PRIMARY KEY,
  tenant_id BIGINT NOT NULL,
  username VARCHAR(64) NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  real_name VARCHAR(64) NOT NULL,
  department_id BIGINT,
  status VARCHAR(20) NOT NULL DEFAULT 'ENABLED',
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  deleted TINYINT NOT NULL DEFAULT 0,
  UNIQUE KEY uk_tenant_username (tenant_id, username),
  KEY idx_tenant_department (tenant_id, department_id)
);
```

字段说明：

| 字段 | 说明 |
|---|---|
| `tenant_id` | 租户 ID |
| `username` | 登录名 |
| `password_hash` | 密码哈希 |
| `department_id` | 所属部门 |
| `status` | `ENABLED` / `DISABLED` |
| `deleted` | 软删除 |

## 8.2 角色表

```sql
CREATE TABLE sys_role (
  id BIGINT PRIMARY KEY,
  tenant_id BIGINT NOT NULL,
  role_code VARCHAR(64) NOT NULL,
  role_name VARCHAR(64) NOT NULL,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  deleted TINYINT NOT NULL DEFAULT 0,
  UNIQUE KEY uk_tenant_role_code (tenant_id, role_code)
);
```

## 8.3 权限表

```sql
CREATE TABLE sys_permission (
  id BIGINT PRIMARY KEY,
  permission_code VARCHAR(128) NOT NULL,
  permission_name VARCHAR(128) NOT NULL,
  module_code VARCHAR(64) NOT NULL,
  created_at DATETIME NOT NULL,
  UNIQUE KEY uk_permission_code (permission_code)
);
```

## 8.4 用户角色关联表

```sql
CREATE TABLE sys_user_role (
  id BIGINT PRIMARY KEY,
  tenant_id BIGINT NOT NULL,
  user_id BIGINT NOT NULL,
  role_id BIGINT NOT NULL,
  created_at DATETIME NOT NULL,
  UNIQUE KEY uk_user_role (tenant_id, user_id, role_id),
  KEY idx_role_id (role_id)
);
```

## 8.5 角色权限关联表

```sql
CREATE TABLE sys_role_permission (
  id BIGINT PRIMARY KEY,
  tenant_id BIGINT NOT NULL,
  role_id BIGINT NOT NULL,
  permission_id BIGINT NOT NULL,
  created_at DATETIME NOT NULL,
  UNIQUE KEY uk_role_permission (tenant_id, role_id, permission_id),
  KEY idx_permission_id (permission_id)
);
```

## 8.6 部门表

```sql
CREATE TABLE sys_department (
  id BIGINT PRIMARY KEY,
  tenant_id BIGINT NOT NULL,
  parent_id BIGINT,
  dept_name VARCHAR(128) NOT NULL,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  deleted TINYINT NOT NULL DEFAULT 0,
  KEY idx_tenant_parent (tenant_id, parent_id)
);
```

## 8.7 知识库访问权限表

```sql
CREATE TABLE kb_space_acl (
  id BIGINT PRIMARY KEY,
  tenant_id BIGINT NOT NULL,
  space_id BIGINT NOT NULL,
  subject_type VARCHAR(20) NOT NULL,
  subject_id BIGINT NOT NULL,
  access_level VARCHAR(20) NOT NULL,
  created_at DATETIME NOT NULL,
  UNIQUE KEY uk_space_acl (tenant_id, space_id, subject_type, subject_id, access_level),
  KEY idx_subject (tenant_id, subject_type, subject_id),
  KEY idx_space_level (tenant_id, space_id, access_level)
```

枚举说明：

| 字段 | 可选值 |
|---|---|
| `subject_type` | `USER` / `ROLE` / `DEPARTMENT` |
| `access_level` | `READ` / `MANAGE` |

## 10. 资源权限判断 SQL 示例

## 10.1 查询用户权限码

```sql
SELECT DISTINCT p.permission_code
FROM sys_user u
JOIN sys_user_role ur
  ON ur.tenant_id = u.tenant_id
 AND ur.user_id = u.id
JOIN sys_role r
  ON r.tenant_id = ur.tenant_id
 AND r.id = ur.role_id
 AND r.deleted = 0
JOIN sys_role_permission rp
  ON rp.tenant_id = r.tenant_id
 AND rp.role_id = r.id
JOIN sys_permission p
  ON p.id = rp.permission_id
WHERE u.tenant_id = ?
  AND u.id = ?
  AND u.status = 'ENABLED'
  AND u.deleted = 0;
```

## 10.2 判断知识库读取权限

```sql
SELECT 1
FROM kb_space_acl acl
WHERE acl.tenant_id = ?
  AND acl.space_id = ?
  AND acl.access_level IN ('READ', 'MANAGE')
  AND (
    (acl.subject_type = 'USER' AND acl.subject_id = ?)
    OR (acl.subject_type = 'ROLE' AND acl.subject_id IN (
      SELECT role_id FROM sys_user_role WHERE tenant_id = ? AND user_id = ?
    ))
    OR (acl.subject_type = 'DEPARTMENT' AND acl.subject_id = ?)
  )
LIMIT 1;
```

## 10.3 判断文档读取权限

```sql
SELECT 1
FROM kb_document_acl acl
WHERE acl.tenant_id = ?
  AND acl.document_id = ?
  AND acl.access_level IN ('READ', 'MANAGE')
  AND (
    (acl.subject_type = 'USER' AND acl.subject_id = ?)
    OR (acl.subject_type = 'ROLE' AND acl.subject_id IN (
      SELECT role_id FROM sys_user_role WHERE tenant_id = ? AND user_id = ?
    ))
    OR (acl.subject_type = 'DEPARTMENT' AND acl.subject_id = ?)
  )
LIMIT 1;
```

## 11. 审计要求

以下操作必须写审计日志：

1. 登录成功、登录失败、退出登录。
2. 上传文档。
3. 删除文档。
4. 修改文档权限。
5. 构建文档索引。
6. 重试索引任务。
7. 全量重建索引。
8. 导出审计日志。

建议审计字段：

```text
tenant_id
user_id
username
action
resource_type
resource_id
request_uri
request_method
client_ip
result
error_msg
created_at
```

## 12. 后端实现注意事项

1. 所有业务查询必须带 `tenant_id`。
2. 所有写接口必须同时校验系统功能权限和资源数据权限。
3. 401 和 403 必须区分。
4. `/api/auth/me` 返回的权限必须和前端 `permission` 字符串完全一致。
5. 问答接口不能只校验登录，必须校验用户是否有该知识库访问权。
6. 引用详情必须校验对应文档读取权限。
7. 文档权限变更后，建议清理相关权限缓存。
8. 删除角色、调整权限后，建议清理用户权限缓存。
9. 全量重建、删除文档、批量重试必须写审计日志。
10. 不允许前端传 `tenantId` 后直接信任，应以后端 token 中的租户为准。

## 13. 联调验收清单

1. 登录后 `/api/auth/me` 返回 `permissions`。
2. 普通用户没有 `document:manage` 时，前端不展示文档管理菜单。
3. 普通用户直接访问 `/api/kb/documents` 时，后端返回 403。
4. 管理员访问 `/api/kb/documents` 正常返回分页数据。
5. 没有知识库访问权的用户请求 `spaceId` 时返回 403。
6. 没有文档读取权的用户请求引用详情时返回 403。
7. token 过期时接口返回 401。
8. 文档上传、删除、索引重试、全量重建均有审计记录。
9. 角色新增权限后，重新登录或刷新用户信息可获得新权限。
10. 角色移除权限后，用户不能继续访问对应接口。
